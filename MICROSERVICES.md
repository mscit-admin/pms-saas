# بنية الخدمات المصغّرة (Microservices)

حُوّل التطبيق من تطبيق Next.js أحادي (monolith) إلى ثلاث خدمات مستقلّة قابلة
للنشر والتوسعة كلٌّ على حدة، تتشارك قاعدة بيانات MySQL واحدة، وتُنسَّق عبر
Docker Compose.

```
                       ┌──────────────┐
   المتصفح  ──────────▶│   web (Next) │  المنفذ 3000
                       │  واجهة فقط   │
                       └──────┬───────┘
                   /api/*  إعادة كتابة (proxy)
                              ▼
                       ┌──────────────┐        ┌──────────────┐
                       │  api (Express)│◀──────│ worker (Node)│
                       │ كل مسارات /api│       │ مزامنة دورية │
                       └──────┬───────┘        └──────┬───────┘
                              │                       │
                              ▼                       ▼
                         ┌───────────────────────────────┐
                         │        MySQL (db)             │
                         └───────────────────────────────┘
```

## الخدمات

| الخدمة | المسار | التقنية | الدور | المنفذ |
|---|---|---|---|---|
| **web** | `services/web` | Next.js 14 | الواجهة (React) فقط؛ تعيد كتابة `/api/*` إلى خدمة الـ API | 3000 |
| **api** | `services/api` | Express | تخدم كل مسارات `/api` (88 مساراً) فوق المنطق المشترك | 3001 |
| **worker** | `services/worker` | Node | حلقة المزامنة الدورية مع جيرا (Polling) | — |
| **core** | `packages/core` | حزمة مشتركة | قاعدة البيانات، المصادقة، جيرا، التحليلات، المخطط | — |

## كيف تعمل خدمة الـ API دون إعادة كتابة المعالجات

معالجات `route.js` نُقلت **كما هي** من `src/app/api` إلى `services/api/app/api`،
وتُشغَّل في Express عبر طبقتين رفيعتين:

- **خطّاف دقّة ESM** (`src/loader.mjs`): يحوّل `@/lib/*` إلى الحزمة `@pms/core`،
  و`next/server` إلى بديل خفيف فوق `Response` القياسي.
- **موجّه نظام الملفات** (`src/fs-router.js`): يكتشف كل `route.js` ويُسجّلها
  تلقائياً بنفس اصطلاح Next (`[key]` ← `:key`)، مع تقديم المسارات الثابتة
  على المتغيّرة.
- **المُكيِّف** (`src/adapter.js`): يبني `Request` من طلب Express، يُشغّل المعالج
  داخل سياق كوكي مستقلّ (`@pms/core/cookies`)، ثم يحوّل `Response` الناتج إلى
  ردّ Express مع الكوكيات.

بذلك يبقى منطق المعالجات والتحليلات بلا تغيير، وتنعدم احتمالية انحراف السلوك.

### ما الذي تغيّر في الشيفرة المشتركة

- `packages/core/src/http.js`: يُعيد `Response` القياسي بدل `NextResponse`.
- `packages/core/src/auth.js`: قراءة/كتابة الكوكي عبر `cookies.js`
  (`AsyncLocalStorage`) بدل `next/headers`.
- `services/web/src/app/layout.js`: يجلب بيان الهوية من الـ API عبر HTTP
  (تبقى خدمة الواجهة بلا اتصال مباشر بقاعدة البيانات).

## التشغيل

```bash
cp .env.docker.example .env       # ثم املأ الأسرار وكلمة مرور القاعدة
docker compose up -d --build      # db + migrate + api + worker + web

# أول مرّة فقط: إنشاء مستخدم المدير
docker compose run --rm api node /app/packages/core/scripts/seed-admin.js
```

الواجهة على `http://localhost:3000`. خدمة الهجرة `migrate` تطبّق المخطط ثم تخرج،
وتنتظرها `api` و`worker` قبل الإقلاع.

## التطوير محلياً (دون Docker)

```bash
npm install                # مساحات عمل npm (workspaces)
npm run db:migrate         # تطبيق المخطط (يتطلّب MySQL محلية + .env)
npm run dev:api            # خدمة الـ API على 3001
npm run dev:web            # الواجهة على 3000 (تمرّر /api إلى 3001)
npm run dev:worker         # عامل المزامنة
```

## ملاحظات

- التوسعة الأفقية: يمكن تشغيل عدة نسخ من `api` خلف موازِن أحمال؛ و`worker`
  يُبقى نسخة واحدة (المزامنة الدورية) لتفادي التكرار.
- ملفات `deploy/*.service` (systemd) الخاصة بالنشر الأحادي السابق لم تعُد
  مستخدمة بعد التحوّل إلى Compose، وتُركت للمرجع فقط.
- قاعدة البيانات مشتركة بين الخدمات (قرار مقصود لتقليل المخاطر)؛ ترقية لاحقة
  محتملة: قاعدة لكل خدمة + ناقل أحداث.
